#### Fixed concurrent write conflict in `terragrunt stack generate`

On repositories with nested stack hierarchies, `terragrunt stack generate` could produce non-deterministic filesystem errors — for example, `file exists` or `no such file or directory` errors against files inside `.terragrunt-stack/` — when two worker goroutines emitted into the same target directory at once.

The underlying cause was in the generation dedup map: it was keyed by the raw source stack-file path, but discovery can surface the same physical stack file under multiple string encodings across generation-level iterations (different relative-path bases, unclean path segments, or repeated passes between levels). Two entries with different keys but the same on-disk target would both pass through to the worker pool and step on one another's writes.

Stack-file paths are now canonicalized at the discovery boundary so equivalent paths collapse to a single dedup key. An additional dedup layer keyed by the resolved `.terragrunt-stack/` target directory ensures two distinct stack files never emit into the same subtree concurrently. When a duplicate target is detected, the later source is skipped with a warning that names both paths, surfacing any genuine topology overlap in the user's stack definitions.
