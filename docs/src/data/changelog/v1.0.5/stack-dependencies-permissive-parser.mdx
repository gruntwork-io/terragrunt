#### `stack-dependencies`: bug fixes for parser and stack-level autoinclude values

The `stack-dependencies` experiment ships the following fixes.

The simplified two-pass parser now captures `source`, `path`, and `values` (and include block `path`) as lazy `hcl.Expression` values and evaluates them only when needed against the full Terragrunt eval context built by the production parser. It no longer re-decodes a stack file with a `nil` eval context when any block declares an `autoinclude`. Non-literal expressions in unrelated unit attributes (`${get_repo_root()}/...`, `format(...)`, `local.region`, `values.idps`) no longer block autoinclude resolution.

```hcl
unit "account" {
  source = "${get_repo_root()}/units/account"
  path   = "account"
  values = {
    account = values.account
    region  = local.region
  }
}

unit "idps" {
  source = format("%s/units/idp", get_terragrunt_dir())
  path   = "idps"
  values = { identityProviders = values.idps }
}

unit "roles" {
  source = "${get_repo_root()}/units/roles"
  path   = "roles"

  autoinclude {
    dependency "account" {
      config_path = unit.account.path
    }
  }
}
```

`terragrunt run --all` now walks generated nested stack files via the unified lazy parser. It no longer fails with `Function calls not allowed; Functions may not be called here.` on a `.terragrunt-stack/<name>/terragrunt.stack.hcl` whose unit `source` contains supported terragrunt path functions such as `get_terragrunt_dir()`. Discovery binds the Terraform stdlib plus `get_terragrunt_dir`; other Terragrunt path helpers (`get_repo_root`, `find_in_parent_folders`, `path_relative_to_include`, etc.) are not available in generated nested stack files and now surface a clear evaluation error during discovery instead of being silently dropped.

A `stack { autoinclude { values = {...} } }` block now propagates its resolved values into each generated nested-stack unit's `terragrunt.values.hcl`, so the nested unit's `inputs = { val = values.val }` resolves at runtime instead of failing with `Unknown variable; There is no variable named "values"`. The autoinclude eval context binds each declared dependency's `mock_outputs` so `dependency.X.outputs.Y` references in the values resolve at generation time.

```hcl
stack "stack_w_outputs" {
  source = "../catalog/stacks/stack-w-outputs"
  path   = "stack-w-outputs"
}

stack "stack_w_values" {
  source = "../catalog/stacks/stack-w-values"
  path   = "stack-w-values"

  autoinclude {
    dependency "unit_w_outputs" {
      config_path                             = stack.stack_w_outputs.unit_w_outputs.path
      mock_outputs_allowed_terraform_commands = ["validate", "plan"]
      mock_outputs                            = { val = "mock-val" }
    }

    values = {
      val = dependency.unit_w_outputs.outputs.val
    }
  }
}
```

Internally, the previously duplicated strict and permissive parse paths were unified. The `*Permissive` types and functions in `internal/hclparse/` were removed; the parser now has a single code path that uses lazy expressions and a caller-provided eval context.
