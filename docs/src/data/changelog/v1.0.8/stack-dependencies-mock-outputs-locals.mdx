#### `stack-dependencies`: autoinclude resolves stack-level references at generate time

An `autoinclude` block is evaluated in the context of the `terragrunt.stack.hcl` file where it is declared. Stack-level `local.*`, `unit.<name>.path`, and `stack.<name>.path` resolve at `terragrunt stack generate` time and are rendered to literals in the generated `terragrunt.autoinclude.hcl` everywhere, including inside the `inputs` attribute and the `generate` and `remote_state` blocks. Function calls are also evaluated at generate time, except inside those three deferred zones: a function call in `inputs`, `generate`, or `remote_state` is kept verbatim and evaluated inside the generated unit, so directory-sensitive functions (`get_terragrunt_dir`, `find_in_parent_folders`, etc.) resolve in the unit. A `dependency.<name>.outputs.<key>` reference is always kept verbatim and resolved in the unit.

Two patterns are now rejected at generate time with a clear error: a `values.*` reference anywhere in the autoinclude body (values is a unit-scoped namespace that is not available in the stack file; use a stack-level local declared in `terragrunt.stack.hcl` or a literal), and a `locals` block defined inside the autoinclude (declare locals at the stack level instead). A dependency `config_path` may still reference `values.*` for the cross-level path-passing pattern, since `config_path` is fully resolved at generate time.
