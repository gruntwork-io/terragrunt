#### `stack-dependencies`: autoinclude resolves stack-level references at generate time

An `autoinclude` block is evaluated in the context of the `terragrunt.stack.hcl` file where it is declared. Stack-level `local.*`, `unit.<name>.path`, and `stack.<name>.path` resolve at `terragrunt stack generate` time and are rendered to literals in the generated `terragrunt.autoinclude.hcl`, including inside the `inputs`, `generate`, and `remote_state` blocks. Those three blocks are deferred zones: any function call or `dependency.<name>.outputs.<key>` reference there is kept verbatim and evaluated inside the generated unit, so directory-sensitive functions (`get_terragrunt_dir`, `find_in_parent_folders`, etc.) resolve in the unit. Everywhere else (a dependency `config_path`, `mock_outputs`, and other attributes) function calls are evaluated at generate time against the stack file, while any `dependency.<name>.outputs.<key>` reference stays verbatim for the unit.

Two patterns are now rejected at generate time with a clear error: a `values.*` reference anywhere in the autoinclude body (values is a unit-scoped namespace that is not available in the stack file; use a stack-level local declared in `terragrunt.stack.hcl` or a literal), and a `locals` block defined inside the autoinclude (declare locals at the stack level instead). A dependency `config_path` may still reference `values.*` for the cross-level path-passing pattern, since `config_path` is fully resolved at generate time.
