## Provision remote_state

Using this command bootstraps the resources described in your [`remote_state` block](/reference/hcl/blocks/#remote_state).

If any of the resources described in the `remote_state` block need provisioning, `bootstrap` will provision them. If they are present, but configured in a way that differs from `remote_state` configuration, Terragrunt will attempt to update them when it is safe to do so.

For example, if you have the following `remote_state` block:

```hcl
# terragrunt.hcl

remote_state {
  backend = "s3"
  config = {
    bucket  = "mybucket"
    key     = "path/to/my/key"
    region  = "us-east-1"
    encrypt = true

    dynamodb_table = "tf-lock"

    accesslogging_bucket_name = "mybucket-logs"
  }
}
```

Then run the following:

```bash
terragrunt backend bootstrap
```

You'll ensure the availability of:

- An S3 bucket named `mybucket` in the `us-east-1` region with the following enabled:
  - Server Side Encryption (SSE)
  - Versioning
  - TLS Enforcement
- A DynamoDB table named `tf-lock` in the `us-east-1` region with SSE.
- An S3 bucket named `mybucket-logs` configured as the access log destination for the `mybucket` bucket.

The `bootstrap` command is idempotent. If the resources already exist, `bootstrap` will not provision them again.

<Aside type="tip" title="--backend-bootstrap">

The flag `--backend-bootstrap` is equivalent to explicitly running the `bootstrap` command.

Using it in conjunction with any `run` command will result in any required bootstrapping to be performed prior to initiating the run.

</Aside>

## Azure Backend

<Aside type="caution" title="Experimental">

Bootstrap support for the `azurerm` backend is gated behind the [`azure-backend`](/reference/experiments/active#azure-backend) experiment. Enable it with `--experiment azure-backend` (or `experiments = ["azure-backend"]` in the unit config). Without the experiment, `terragrunt backend bootstrap` is a no-op for `azurerm` units.

</Aside>

Given the following `remote_state` block:

```hcl
# terragrunt.hcl

remote_state {
  backend = "azurerm"
  config = {
    storage_account_name = "mystateacct"
    container_name       = "tfstate"
    key                  = "path/to/my/key.tfstate"
    resource_group_name  = "my-rg"
    subscription_id      = "00000000-0000-0000-0000-000000000000"

    # Terragrunt-only fields (filtered out before terraform init)
    location                            = "westeurope"
    create_storage_account_if_not_exists = true
    enable_versioning                   = true
    use_azuread_auth                    = true
  }
}
```

Then running:

```bash
terragrunt --experiment azure-backend backend bootstrap
```

ensures the availability of:

- A resource group named `my-rg` in the `westeurope` region (created when `create_storage_account_if_not_exists = true` and the group is missing).
- A storage account named `mystateacct` with blob versioning and a 7-day blob soft-delete window enabled.
- A blob container named `tfstate`.
- When `use_azuread_auth = true`, the calling principal is granted the `Storage Blob Data Owner` role on the container.

State locking on Azure is handled by the native `azurerm` backend via blob leases — Terragrunt does not provision a separate lock resource (no Azure equivalent to S3's DynamoDB table is required).

The `bootstrap` command is idempotent: re-running it against a fully provisioned account is a no-op.
